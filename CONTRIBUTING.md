# Contributing to Semantica

Thank you for your interest in contributing to Semantica. This guide covers how to set up your development environment and submit changes.

## Prerequisites

- Go 1.26+
- [sqlc](https://docs.sqlc.dev/en/latest/overview/install.html) v1.30+ (for regenerating SQL code)
- [golangci-lint](https://golangci-lint.run/welcome/install-local/) v2.11.3 (installed via `make lint-install` into `./bin/tools` so local lint matches CI)
- Git

## Setup

```bash
git clone https://github.com/semanticash/cli.git
cd cli
make dep       # download Go modules
make lint-install  # install the pinned golangci-lint version
make build     # compile to ./bin/semantica
```

## Development workflow

### Running tests

```bash
make test          # unit tests
make test-race     # unit tests with race detector
make test-e2e      # end-to-end tests (builds binary first)
make lint          # pinned golangci-lint v2.11.3 (same checks as CI)
```

### Code generation

The SQLite query layer is generated by [sqlc](https://sqlc.dev). If you modify files in `internal/store/sqlite/queries/` or `internal/store/sqlite/schema/`:

```bash
make generate          # regenerate Go code from SQL
make check-generated   # verify generated code is up to date
```

CI runs `check-generated` on every PR - if it fails, run `make generate` and commit the result.

### Project structure

```
cmd/semantica/          # CLI entrypoint
internal/
  commands/             # cobra command definitions
  service/              # core business logic (worker, rewind, explain, etc.)
  store/sqlite/         # SQLite storage layer (sqlc-generated)
  git/                  # Git operations (hooks, diff, log)
  hooks/                # AI provider hook integrations (Claude, Cursor, Gemini, Copilot)
  broker/               # cross-repo event routing
  version/              # build version injection
e2e/                    # end-to-end tests (build-tag gated)
```

See [docs/architecture.md](docs/architecture.md) for a deeper walkthrough.

### Repository boundary

This repository contains the CLI, local storage, hook capture, attribution engine, and MCP server. Hosted features - GitHub PR comments and check runs, GitLab MR comments and commit statuses, and dashboards - live in the Semantica backend/API repository.

## Submitting changes

1. Fork the repository and create a branch from `main`.
2. Make your changes. Add or update tests as appropriate.
3. Run `make test-race` and `make lint` locally.
4. If you changed SQL queries or schema, run `make generate` and commit the output.
5. Open a pull request against `main`.

### PR guidelines

- Keep PRs focused - one logical change per PR.
- Write a clear description of what the change does and why.
- All CI checks must pass: unit tests, lint, generated code check, cross-compile, E2E tests, and shellcheck.
- GitHub Copilot PR review may add advisory comments based on `.github/copilot-instructions.md`.
- Additional GitHub security workflows are configured for dependency review and CodeQL analysis.

### Commit messages

Use a short imperative subject line (e.g., "Add cursor transcript ingestion", "Fix rewind safety checkpoint race"). No strict format is enforced, but keep messages descriptive.

## Code style

- Follow standard Go conventions (`gofmt`, `go vet`).
- golangci-lint enforces the project's lint rules - run `make lint` before pushing.
- `make lint` uses a repo-local pinned golangci-lint binary in `./bin/tools`, installed by `make lint-install`.
- Avoid adding dependencies unless clearly justified.

## Testing

- **Unit tests** (`go test ./...`): test individual packages in isolation. Use real SQLite databases (not mocks) for store tests.
- **E2E tests** (`go test -tags e2e ./e2e/...`): test the compiled binary end-to-end in isolated temp repos. These require `make build` first.
- If your change touches hook behavior, provider ingestion, or CLI output, add or update E2E tests.

## Reporting issues

Open an issue on [GitHub](https://github.com/semanticash/cli/issues). Include:

- Semantica version (`semantica --version`)
- OS and architecture
- Steps to reproduce
- Expected vs actual behavior
